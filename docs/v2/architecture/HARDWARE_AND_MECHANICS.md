# Hardware And Mechanics — V2

## Purpose

This document collects the physical robot context that supports the V2 firmware
architecture: board responsibilities, power topology, communication wiring,
actuator layout, and the mechanical model assumed by the motion stack.

For schematic-level detail — net tables, footprints, known ERC issues — see
[`hardware/v2/README.md`](../../../hardware/v2/README.md).

---

## Hardware Summary

V2 introduces three dedicated PCBs replacing the single board of V1.

| Board | MCU | Role |
|-------|-----|------|
| MainBoard | ESP32 | Motion control, IMU, RS485 master, logic power distribution |
| LegBoard (×6) | XIAO RP2040 | Local servo PWM, current sensing, RS485 slave |
| MainPowerBoard | — | Battery input, fusing, UBEC/SBEC modules, servo power distribution |

Key differences from V1:
- Servo PWM is no longer generated by the ESP32. Each LegBoard generates PWM
  locally for its three servos.
- Communication between MainBoard and LegBoards is over a shared RS485 bus
  carried by RJ11 cables.
- Per-servo current is measured on each LegBoard via dedicated sense amplifiers
  and reported back to the ESP32.
- High-current servo loads are isolated from the MainBoard at the hardware
  level.

---

## Board Responsibilities

### MainBoard

Central logic and communication controller.

- Hosts the ESP32: gait generation, inverse kinematics, body stabilisation, IMU
  processing, controller input, RS485 master.
- Hosts the IMU. All orientation processing is centralised here; LegBoards have
  no orientation awareness.
- SP3485 RS485 transceiver with direction control on ESP32 IO4 (`RS485_DE`).
- Six RJ11 leg interfaces (one per leg), carrying logic power and RS485.
- Receives SBEC 5 V logic supply and distributes it to all LegBoards via RJ11.
- Does not carry servo current.

Firmware-relevant pin assignments:

| Signal | ESP32 pin |
|--------|-----------|
| RS485 TX (UART2) | IO17 |
| RS485 RX (UART2) | IO16 |
| RS485 DE | IO4 |
| IMU SDA | IO21 |
| IMU SCL | IO22 |
| IMU INT | IO34 |

### LegBoard

One per leg. Intelligent local actuator controller.

- Hosts a Seeed XIAO RP2040.
- SP3485 RS485 transceiver; direction control on XIAO P6/D4 (`RS485_DE`).
- Receives commands over RS485, generates servo PWM locally for coxa, femur,
  and tibia joints.
- INA4181 4-channel current sense amplifier: one channel per servo plus a
  total-leg channel. Current readings are reported back to the ESP32 with each
  telemetry response.
- Watchdog on RS485 communication timeout: holds last commanded position or
  moves to a safe posture if the bus is silent for longer than the configured
  timeout.
- Performs local trajectory interpolation between received motion targets,
  producing smooth servo motion at a higher rate than the RS485 command rate.

Firmware-relevant pin assignments (XIAO RP2040):

| Signal | XIAO pin |
|--------|----------|
| RS485 TX (UART) | P0 |
| RS485 RX (UART) | P1 |
| RS485 DE | P6 / D4 |
| Coxa PWM | P3 |
| Femur PWM | P4 |
| Tibia PWM | P2 |
| Total current sense | A0 |
| Coxa current sense | A1 |
| Femur current sense | A2 |
| Tibia current sense | A3 |

### MainPowerBoard

Responsible for all high-current power distribution. No MCU.

- Battery input with main fuse, reverse-polarity protection (PMOS), and master
  power switch.
- Three UBEC modules: servo power distributed in pairs to the six LegBoards.
- One SBEC module: 5 V logic supply to MainBoard.
- Bulk capacitance per UBEC output to absorb transient servo loads.
- Monitor header exposing battery voltage and all UBEC output voltages.
- Status LEDs: UBEC 1/2/3 present, SBEC present.

## Power Architecture

Logic and servo power are completely separated.

### Logic Power Path

```
LiPo → MainPowerBoard SBEC → MainBoard +5V rail → RJ11 pin 5 → LegBoard XIAO VBUS
```

Powers: ESP32, IMU, SP3485 transceivers on all boards, XIAO RP2040s, INA4181
sense amplifiers.

### Servo Power Path

```
LiPo → MainPowerBoard → UBEC 1 → LegBoards 1 and 2
                      → UBEC 2 → LegBoards 3 and 4
                      → UBEC 3 → LegBoards 5 and 6
```

Servo power enters each LegBoard on a dedicated 2-pin screw terminal and is
distributed through the on-board current shunts to the three servo connectors.
Servo current never passes through the MainBoard.

**Architectural rule: servo current must never flow through the MainBoard.**
MainBoard carries logic power, RS485 signals, and IMU signals only.

## Inter-Board Connections

### RJ11 Leg Link (MainBoard ↔ LegBoard)

Physical cable: 4-wire link inside a 6P6C / RJ25 shell. Pins 1 and 6 are
unpopulated.

| Pin | Net | Function |
|-----|-----|----------|
| 1 | — | unused |
| 2 | GND | Ground |
| 3 | RS485\_A | RS485 differential A |
| 4 | RS485\_B | RS485 differential B |
| 5 | +5V | Logic power to LegBoard |
| 6 | — | unused |

One RJ11 cable per leg. The RS485 bus is a multi-drop line shared across all
six legs; all six LegBoards are addressable from the single SP3485 transceiver
on the MainBoard.

### Servo Power (MainPowerBoard ↔ LegBoard)

2-pin screw terminal on each LegBoard. Carries the regulated UBEC output
voltage (nominally 6 V) to that leg's servo bus.

### Logic Output (MainPowerBoard ↔ MainBoard)

4-pin header on the MainPowerBoard (`J_LOGIC`) to the MainBoard SBEC input
(`J7`). Carries the 5 V SBEC output.

## Control Frequencies

| Subsystem | Rate | Responsibilities |
|-----------|------|-----------------|
| MainBoard (ESP32) | ~100 Hz | IMU update, gait generation, IK, trajectory generation, RS485 command dispatch |
| LegBoard (RP2040) | ~500–1000 Hz | Local interpolation, servo PWM update, current sampling, watchdog |

The RP2040 internal loop rate is independent of the RS485 command rate. It
interpolates between received motion targets at its own rate to produce smooth
servo motion, tolerating short gaps in RS485 communication without visible
stepping.

---

## Mechanical Layout

Unchanged from V1. Each leg has three joints:

1. **Coxa** — yaw and horizontal reach.
2. **Femur** — main pitch lift motion.
3. **Tibia** — distal extension and stance height control.

The robot uses a custom 3D-printed frame and leg structure sized around
standardised servo dimensions and the target stance geometry.

---

## Nominal Link Geometry

Unchanged from V1.

| Link | Length |
|------|--------|
| Coxa | 0.068 m |
| Femur | 0.088 m |
| Tibia | 0.127 m |

These dimensions define the nominal leg model used by inverse kinematics and
related runtime configuration.

---

## Coordinate Conventions

Unchanged from V1.

- Body frame: `+X` forward, `+Y` left, `+Z` up.
- Leg-local planning frame: leg mount transforms rotate body-frame targets into
  each leg's local coordinate system before IK is solved.

The exact per-leg mount poses are part of the robot configuration domain and
are consumed by the motion stack at runtime.

## Why This Matters For Firmware

**No PWM peripheral budget constraint on the ESP32.** V1 required mixing MCPWM
and LEDC to produce 18 servo signals. In V2 the ESP32 produces no servo PWM;
all 18 signals are generated locally on the six LegBoards.

**RS485 bus latency budget shapes the comm task design.** The `hex_rs485_master`
task sends and polls each leg sequentially. Per-leg transaction time (baud rate,
frame sizes, DE turnaround, RP2040 response latency) determines the telemetry
refresh rate and must be sized so that current data is fresh enough for
touchdown detection. See [`../interfaces/RS485_PROTOCOL.md`](../interfaces/RS485_PROTOCOL.md) for the protocol
design.

**Per-servo current is the primary ground-contact signal.** The INA4181 on each
LegBoard measures current through each servo independently. A current spike on
the tibia (or femur) servo is the intended proxy for leg touchdown. This data
is returned in the telemetry response of every RS485 transaction and stored in
the `hex_rs485_master` telemetry buffer for use by the locomotion stack.

**RP2040 interpolation decouples servo smoothness from RS485 rate.** The
LegBoard does not require a new command every servo PWM cycle. It interpolates
locally between received targets, so RS485 bandwidth can be traded against
servo command granularity without visible motion degradation.

**Watchdog behavior is a safety boundary.** If the RS485 bus goes silent, each
LegBoard independently holds its last commanded position or transitions to a
safe posture after a configured timeout. This means a firmware crash or bus
fault on the ESP32 does not leave servos uncontrolled.
